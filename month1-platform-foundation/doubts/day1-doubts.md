# Day 1 — Doubts & Open Questions

**Date:** 2026-03-25  
**Rule:** Every question here must be precise enough that a concrete answer can be given. No vague questions like "how does Kubernetes work". Every doubt should expose a specific gap that, if filled, would change how I debug something.

## Doubt 1 — Why does a pod pass liveness but fail readiness?

**What I think I know:**
Both are HTTP probes (or exec, TCP) that kubelet runs against the pod. Liveness failure restarts the container. Readiness failure removes the pod from the Service's endpoint list.

**The specific gap:**
If both probes are configured and the container process is the same — why would one return 200 and the other not?

**What I suspect:**
They must be hitting different endpoints (`/healthz` vs `/ready`) that check different things. But what different things? And is this always true, or can a pod be alive but not ready for a reason that doesn't involve a different endpoint?

**What I need to understand:**
- The exact sequence of events: when does kubelet start running each probe?
- Can a pod fail readiness while passing liveness if they hit the **same** endpoint? Under what conditions?
- What does "not ready" actually block — just Service traffic, or also Ingress routing? Both?
- If I have 3 pods and 1 fails readiness, do all 3 remain "running" in `kubectl get pods`? Does the Deployment try to replace it?

**Why this matters in production:**
If I see a pod that is `Running` (1/1) but receives no traffic, I need to know immediately whether it's a readiness issue or an endpoints issue (which is downstream of readiness). I currently cannot confidently trace this chain without guessing.

## Doubt 2 — How does kube-proxy choose between endpoints when a Service has multiple pods?

**What I think I know:**
kube-proxy maintains iptables rules (or IPVS rules) that DNAT traffic from the ClusterIP to one of the pod IPs. The selection is round-robin by default.

**The specific gap:**
In iptables mode, how is "round-robin" implemented? iptables is stateless and evaluates rules top-to-bottom. It doesn't have a round-robin algorithm built in. So how does it distribute traffic across multiple endpoints?

**What I suspect:**
There is a `statistic` module in iptables that uses probability math — the first rule matches with 1/N probability, the second with 1/(N-1) of remaining, and so on. But I have never confirmed this by reading actual iptables rules generated by kube-proxy.

**What I need to understand:**
- What does the actual iptables rule chain look like for a Service with 3 endpoints?
- If I add a 4th pod to a Service, how quickly do the iptables rules update? Is there a period where the 4th pod is ready but not yet receiving traffic?
- In IPVS mode, does this work differently? Is IPVS mode strictly better, and if so, why isn't it the default?
- What happens to in-flight connections when a pod is removed (readiness fails)? Are existing TCP connections drained or abruptly closed?

**Why this matters in production:**
I once saw uneven traffic distribution across pods — some pods at 80% CPU, others at 10%. I assumed it was an application issue. Could it have been an iptables/IPVS distribution issue? I don't know.

## Doubt 3 — When does DNS TTL cause stale routing inside a cluster?

**What I think I know:**
DNS records have TTLs. After a TTL expires, clients must re-query. Before expiry, the cached IP is used even if the actual target has changed.

**The specific gap:**
Inside a Kubernetes cluster, when does stale DNS cause a real problem?

Kubernetes Service ClusterIPs are stable — they don't change after creation. So why would DNS TTL matter? The IP never changes.

But: what about pod-to-pod communication where the app is using a pod's hostname directly (like a StatefulSet pod)? Or what about external DNS entries that change during a failover?

**What I need to understand:**
- What is the DNS TTL for `my-svc.default.svc.cluster.local` in CoreDNS? Is it configurable?
- If I delete a Service and recreate it with the same name, does it get the same ClusterIP? If not, clients with the old IP cached will fail until TTL expires — is this a real scenario?
- For StatefulSets (`pod-0.my-svc.default.svc.cluster.local`), what is the TTL? Does it change when a pod is replaced?
- How does Java's JVM DNS caching interact with CoreDNS TTLs? (JVM caches DNS indefinitely by default unless `networkaddress.cache.ttl` is set — this is a known footgun)

**Why this matters in production:**
I want to understand when a DNS-layer problem manifests as an application connection error that I might otherwise blame on the network layer.

## Doubt 4 — What is the precise difference between a 502 and 504 from nginx Ingress's perspective?

**What I think I know:**
Both involve the upstream (the backend pod). 502 is "bad gateway" and 504 is "gateway timeout."

**The specific gap:**
I can recite the definitions. I cannot confidently distinguish them from nginx logs alone. What exactly does nginx log differently for a 502 vs a 504?

**What I need to understand:**
- When nginx returns 502: what exactly happened? Did the upstream send back a malformed HTTP response? Did the TCP connection succeed but the HTTP response was invalid? Did the upstream close the connection mid-response?
- When nginx returns 504: what exactly happened? Did nginx wait for `proxy_read_timeout` and get no bytes back? Or did the upstream send a partial response and then stall?
- In the nginx error log, what is the exact log line for a 502 vs a 504? What fields distinguish them?
- If a pod crashes mid-request (OOMKilled while processing), does nginx return 502 or 504 to the client?
- Can a 502 be caused by a misconfigured nginx annotation (e.g., wrong `backend-protocol`)? What would that look like?

**Why this matters in production:**
When I see a 5xx in my alerting, the first thing I do is check nginx logs. I need to read those logs and immediately know what layer is implicated — 502 means "the upstream was reachable but broke the connection" (Layer 6/7), 504 means "the upstream was reachable but too slow" (Layer 7/8). Getting these wrong wastes time in an incident.

## Doubt 5 — How does the startup probe interact with the liveness and readiness probes?

**What I think I know:**
The startup probe, if configured, must succeed before liveness and readiness probes begin running. It is designed for slow-starting containers (JVM apps, containers that need to load large models) where a high `initialDelaySeconds` on liveness would be needed otherwise.

**The specific gap:**
What exactly is "gated"? If the startup probe is failing, is the pod:
- Not yet included in Service endpoints (readiness not started)?
- Subject to restart (liveness not started)?
- Both?
- Neither — it just sits in a "starting" limbo?

And: if I configure a startup probe with `failureThreshold: 30` and `periodSeconds: 10`, the pod gets 300 seconds to start before kubelet kills it. What state is the pod in during those 300 seconds? `Running`? `Pending`? Does it show `0/1` in `kubectl get pods`?

**Why this matters in production:**
I have seen pods stuck for 5 minutes showing `Running 0/1 Ready` during startup. I assumed this was a readiness probe issue and tried to investigate the app. If it was actually a startup probe, I was looking in the wrong place with the wrong expectation.

## Doubt 6 — What actually happens to the nginx configuration when I add a second Ingress resource?

**What I think I know:**
The nginx Ingress controller watches all Ingress resources across namespaces and generates a single `nginx.conf`. It creates `server` blocks per host and `location` blocks per path.

**The specific gap:**
If I have two Ingress resources — one in namespace `frontend` routing `/` to Service A, and one in namespace `api` routing `/api` to Service B — both for the same host `app.example.com` — what happens?

- Do they merge into one `server { ... }` block in nginx.conf?
- Or do they conflict and one wins?
- If they merge, what is the precedence rule for overlapping `location` blocks?
- What if I have `pathType: Prefix` for `/api` and `pathType: Prefix` for `/` — which one catches `/api/v1/users`?

**Why this matters in production:**
In a real organization, different teams often own different Ingress resources for the same domain. If I don't understand how they interact in the generated nginx config, I cannot debug routing surprises when a new team adds an Ingress and breaks another team's routing.

## Real Incident — 2026-03-27 (Resolved)

**Symptom:** `http://configserver.c-qa4.svc.rnq.k8s.copart.com/` returning 404. Direct IP curl returning 401.

**Investigation steps:**
1. `curl http://10.148.14.190:8080` → 401 (app alive, Layer 7 healthy)
2. `telnet 10.148.14.190 8080` → connected (Layer 3 healthy)
3. `kubectl get ingress -n c-qa4` → no resource found (no K8s Ingress — org uses external nginx)
4. `kubectl describe svc configserver -n c-qa4` → type ClusterIP, no external IP
5. Read nginx config → external nginx VM routes by Host header via `map` block
6. `curl -v -H "Host: c-config-qa4.copart.com" http://configserver.c-qa4.svc.rnq.k8s.copart.com/` → 401 ✓

**Root cause:** nginx server_name only matches `c-config-qa4.copart.com`. The internal K8s DNS name was used as the public URL — nginx had no matching server block → returned 404. App was never involved.

**Layer breakdown:**
- Layer 3 — Healthy (telnet connected)
- Layer 4 — Broken (wrong Host header, nginx server_name mismatch)
- Layer 5 — Healthy (ClusterIP reachable from nginx)
- Layer 7 — Healthy (401 = app running, Spring Security responding)

**Key learning:**
- nginx routes by Host header, not just IP/port — this is virtual hosting
- 404 from nginx ≠ app is broken — it means nginx found no matching server block
- The internal K8s DNS name (`svc.cluster.local`) is the upstream backend address, NOT the public hostname
- Always check: what hostname is nginx `server_name` expecting before assuming app is down
- Your org uses external nginx VM + BGP instead of Kubernetes Ingress object

**Correct public URL:** `http://c-config-qa4.copart.com/`
